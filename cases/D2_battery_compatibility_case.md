# Case Study: 배터리 BMS 정합성 검증

> 프로젝트: PLUVA FRIEND SS500 (GT-SS500)
> 역할: 전력제어개발팀 선임연구원 (이슈 발행 및 검증 주도)
> 기간: 2025.12 ~ 2026.02 (3개월, 진행 중)
> 이슈 번호: ISSUE #72 (최우선), #78, #000

---

## Situation — 상황

SS500은 48V 460Ah LiFePO4 배터리(22.08kWh, 15S2P 구성)를 탑재한 전동 농기계입니다. 배터리는 외부 업체가 공급하며, VCU(Vehicle Control Unit)와 CAN 버스로 통신하여 SOC(State of Charge), 셀 전압, 온도, 보호 상태를 실시간 모니터링합니다.

Alpha 3호기 + Pilot 2호기가 제작된 시점에서, 배터리 BMS의 CAN 정합성과 보호 로직 검증이 양산 전 필수 관문이었습니다.

**왜 이게 중요한가:**
- 48V 460Ah는 상당한 에너지(22kWh) — 보호 로직 미작동 시 화재/폭발 위험
- 농기계는 비전문가(농민)가 운용 — 경고 표시가 정확해야 함
- 양산 16대 + Spare 4대가 3월 출하 예정 — 배터리 품질이 양산 일정을 좌우

---

## Task — 과제

**핵심 질문**: "배터리 업체가 납품한 BMS의 보호 로직이 실제로 동작하는가?"

구체적 검증 항목:
1. **CAN DB 정합성**: 배터리 업체가 정의한 CAN 신호가 우리 VCU와 맞는가?
2. **SOC 정확도**: 충전/방전 시 SOC가 정확하게 표시되는가?
3. **보호 로직**: 과충전/과방전/셀 불균형 시 Warning → Fault → Protection이 순차 발동하는가?
4. **Kill Switch**: 비상정지 시 배터리가 안전하게 Shutdown되는가?
5. **Power Hold**: VCU가 Shutdown 요청 시 릴레이가 정상 차단되는가?

**제약 조건**:
- 배터리 업체와 물리적 거리 (합동 시험이 쉽지 않음)
- ROM 업데이트가 무선이 아닌 현장 방문 필요
- Alpha 1~3호기 각각 다른 ROM 버전이 적용된 상태

---

## Action — 행동

### 1차 검증 (2025.12.10) — CAN 정합성 기본 확인

**방법**: Alpha 1호기 납품 배터리의 CAN 로그를 Vector 장비로 수집하여 분석

**발견한 문제**:
- CAN DB가 신규 버전으로 업데이트되었으나, Alpha 1호기에만 적용됨
- 나머지 호기(Alpha 2,3 / Pilot 2~4)는 구형 ROM(Ver 10) 유지
- 배터리 업체 제공 CAN DB와 실측 신호 불일치 항목 존재

**산출물**: `20251210_배터리_및_충전기_정합성_검토_결과.xlsx`

### 2차 검증 (2026.01.14) — 보호 로직 심층 검증

**방법**: Alpha 1호기 ROM 업데이트 후, 완충→방전 전 구간 CAN 로그 수집 + 셀별 전압 분석

**발견한 치명적 문제 3건**:

| # | 문제 | 발생 조건 | 기대 동작 | 실제 동작 | 위험도 |
|---|------|-----------|-----------|-----------|--------|
| 1 | **SOC 오버플로우** | 완충 후 51.9V | SOC 100% + 충전정지 | **SOC 162.4% 표시, Warning/Fault/Protection 미발생** | 치명 |
| 2 | **저 SOC 미경고** | 방전 중 SOC 19% | SOC Warning 발생 | **Warning 미발생** | 높음 |
| 3 | **셀 불균형 미감지** | 셀 편차 260mV (7번 셀 3.34V, 평균 3.46V, 최대 3.60V) | Cell Unbalance Warning/Fault | **미발생** | 높음 |

**데이터 분석 상세**:
```
배터리 팩 전압:     51.9V (완충 상태)
BMS 보고 SOC:       162.4% ← 명백한 오류
셀 평균 전압:        3.46V
7번 셀 전압:        3.34V ← 다른 셀 대비 120mV 낮음
셀 최대 전압:        3.60V
BPA_Cell_DeviV:     평균 150mV, 스파이크 260mV
Warning Flag:       0 (미발생) ← 규격 위반
Fault Flag:         0 (미발생) ← 규격 위반
Protection Flag:    0 (미발생) ← 규격 위반
```

**산출물**: `260114_batt_SOC162_log.asc` (CAN 로그 원본)

### 업체 대응 (2026.01.15~)

**접근 방법**: 감정적 클레임이 아닌, 데이터 기반 요구사항 전달

1. **정량적 이슈 리포트 작성**: 신호별 기대값 vs 실측값 비교표
2. **WBS(Work Breakdown Structure) 수립**: 배터리 업체와 공동 WBS 작성
   - `GINT_SS_농기계_품질_WBS_리스트(20260115)_Updated_GINT.xlsx`
3. **합동 시험 요청**: ROM 업데이트 후 현장 합동 검증 일정 협의 (시급: 1/23)
4. **다중 호기 검증 계획**: Alpha 1,2,3호기 동시 ROM 업데이트 후 비교 검증

### 3차 검증 (2026.02.11~12) — Alpha 1,2,3호기 동시 검증

**방법**: 3개 호기 모두 ROM 업데이트 후 동시 CAN 로그 수집

**검증 항목**:
- Kill Switch 시퀀스 정상 동작 확인
- 보호 로직 구현 상태 검토
- 셀 전압 편차 추적

**산출물**: `20260212_배터리_정합성_검토_결과.xlsx`

### 충전 시스템 연계 이슈 (ISSUE #78)

배터리 정합성과 동시에 충전 시스템 이슈도 발견:

| 문제 | 상세 |
|------|------|
| SOC 변동 | 충전 중 106.5% → 재시동 후 82.2%로 급락 (24.3%p 오차) |
| 충전 전압 변경 | 54V → 52V로 변경됨 (배터리 업체 임의 변경) |
| CAN 통신 에러 | Alpha 1호기 충전 불가 — CAN Communication Error (01/27) |
| 종단저항 | 충전기 종단저항 미제거 상태 (CAN 통신 장애 원인) |
| 릴레이 융착 | 배터리 스위치 접점 릴레이 융착 → 신규 보드 교체 후에도 재발 |

## Result — 결과

### 정량적 성과

| 항목 | Before | After |
|------|--------|-------|
| 보호 로직 검증 | 미검증 (납품 상태 그대로 사용) | **3차 반복 검증 + 신호별 정합성 시트 완성** |
| SOC 정확도 | 162.4% 표시 (오버플로우) | **이슈 정량화 → 업체 ROM 수정 요구** |
| 셀 불균형 감지 | 260mV 편차에도 미감지 | **셀별 전압 모니터링 체계 수립** |
| 업체 관리 | 구두 협의 | **WBS 기반 체계적 일정/품질 관리** |
| 검증 자동화 | 수동 로그 확인 | **CAN 정합성 자동 체크리스트 도입** |

### 비즈니스 임팩트

1. **안전 사고 사전 방지**: SOC 162%에서 보호 로직이 안 도는 상태로 양산에 들어갔으면, 과충전으로 인한 배터리 열폭주 위험이 있었음. LiFePO4는 상대적으로 안전하지만, 462Ah 팩에서 셀 불균형이 누적되면 위험
2. **양산 일정 보호**: 보호 로직 이슈를 양산 전에 발견하여, 양산 후 리콜/필드 이슈로 확대되는 것을 방지
3. **업체 관리 체계화**: 일회성 클레임이 아닌, WBS 기반 지속적 품질 관리 프로세스 수립

### 기술적 학습

- **CAN 정합성 검증은 "한 번"이 아니라 "매 ROM 업데이트마다"** 필요하다는 원칙 확립
- **배터리 업체의 "수정 완료"는 검증하기 전까지 믿지 않는다** — 데이터로 확인
- **다중 호기 동시 검증**의 중요성 — 1호기에서 되는 것이 2호기에서 안 될 수 있음

## 면접 예상 Q&A

**Q: 배터리 업체와 갈등은 없었나요?**

> ROM 업데이트 3차까지 반복하면 업체 관계가 불편해질 수 있습니다. 저는 "왜 안 되냐"가 아니라 "어떤 신호가 기대값과 다르고, 어떤 조건에서 재현된다"는 데이터를 매번 정리해서 전달했습니다. 업체 입장에서도 모호한 클레임보다 명확한 버그 리포트가 더 도움이 됩니다. 결과적으로 WBS를 공동으로 작성하는 수준까지 협력 관계를 발전시켰습니다.

**Q: LiFePO4인데 열폭주 걱정이 과한 거 아닌가요?**

> LiFePO4는 NMC 대비 열적 안정성이 높은 건 맞습니다. 하지만 460Ah 대용량 팩에서 셀 불균형(260mV)이 누적되면, 약한 셀에 과부하가 집중되어 수명 급감 → 내부 단락 위험이 있습니다. 화학적으로 안전한 셀이라도, 시스템 레벨에서 보호 로직은 반드시 검증해야 합니다. 이건 "혹시 몰라서"가 아니라 "기본 안전 설계 원칙"입니다.

**Q: 이 경험을 다른 분야에 어떻게 적용할 수 있나요?**

> 배터리 BMS 정합성 검증의 핵심은 "외부 모듈과 CAN으로 통신하는 시스템에서, 기대 동작과 실제 동작의 차이를 체계적으로 검증하는 것"입니다. 이건 배터리뿐 아니라, 모터 컨트롤러, 센서 모듈, 자율주행 PC — CAN 기반 분산 시스템이라면 어디든 적용됩니다. 특히 로보틱스나 자율주행 차량에서 다수의 ECU가 통신하는 구조에서 이 검증 역량은 직접적으로 가치가 있습니다.
