About Me — 커리어 내러티브

소프트웨어가 물리 세계와 만나는 접점에서 제품을 만드는 사람

한 줄 소개

모터제어 개발자이자 주니어 PM. 하드웨어를 설계하면서 소프트웨어를 이해하고, 기술을 깊이 파면서도 제품 전체를 조망하는 브릿지 엔지니어입니다.

지금까지의 여정

모터제어 개발자로 시작하다

GINT(긴트)의 전력제어개발팀에서 BLDC 모터 제어 개발자로 커리어를 시작했습니다. 첫 프로젝트는 자동차 Tier-1 부품인 SCU(Steering Control Unit) 개발이었습니다. Infineon TC23x MCU 위에서 BLDC 사다리꼴 구동을 구현하고, 자동차 환경시험(ES95400)/EMC(ES96200) 규격을 통과시키는 과정에서 "양산 품질이 무엇인지"를 체감했습니다. DIC(현대) 협력사로서 CAN 통신 설계, PCB 3세대 진화(ES→AS→A2S), SBW 부하시험까지 — 단순히 코드를 짜는 것이 아니라, 제품이 현실 세계에서 동작하도록 만드는 일이었습니다.

이 시기에 직무발명 특허 2건을 출원했습니다: - 모터 초기 위치 검출 알고리즘 (PN231067KR, 2023) — 센서리스 BLDC 구동의 전제 조건인 로터 초기 위치를 추정하는 방법 - 저온 기동 기술 (2024) — 극저온 환경에서 모터 기동 신뢰성을 확보하는 방법

또한 EOP 12V 400W 전동기 국책과제를 수행하며 SVPWM 절환 기법 연구, 온도 조건별(55°/85°/120°) 열평형 시험, 다이나모미터 개발까지 모터 제어의 기술적 깊이를 쌓았습니다.

"제품 전체"를 보기 시작하다

전환점은 PLUVA FRIEND SS500 프로젝트였습니다.

SS500은 농업용 전동 자율주행 Speed Sprayer — 48V 460Ah LiFePO4 배터리로 구동되는 과수원 방제 로봇입니다. 겉으로 보면 농기계지만, 안을 들여다보면:

  • CAN 기반 분산 제어 시스템: VCU(STM32) → 구동모터 × 2 → 팬모터 → 펌프 → BMS, 5개 CAN 노드가 실시간으로 통신
  • 3가지 제어 모드: 리모콘(RC), LCD 패널, 자율주행(ADT) — 각각의 제어권 전환 시 안전을 보장하는 상태머신
  • 자율주행 스택: RTK-GPS(1~2cm 정밀도), 카메라, IMU 기반 경로 추종
  • 클라우드 연동: AWS OTA 업데이트, 작업 데이터 통합 관리

이것은 농기계가 아니라, Physical AI 제품입니다.

저는 이 프로젝트에서 모터제어 개발자(본업)와 주니어 PM(겸임)을 동시에 수행했습니다.

AI를 프로세스에 통합하다

저는 Claude Code를 단순 코딩 보조가 아니라, 개발 프로세스 자체에 통합해서 사용합니다:

  • 5개 에이전트 시스템 설계/운영: PM_Lead, Engineer_1~3, Notion_Admin
  • 이슈 관리 자동화: Notion DB → 로컬 마크다운 동기화 → INDEX 자동 생성
  • 주간보고 자동화: /weekly-report 스킬로 MD + Excel + Notion 동시 출력
  • 기술 분석: CAN 로그 분석, 배터리 정합성 검토를 Claude와 페어로 수행

"AI를 사용하는 수준"이 아니라 "AI 워크플로우를 설계하는 수준"입니다.

왜 Physical AI인가

GINT는 "농기계 회사"가 아닙니다. Foresight(이스라엘 3D 비전) + KONEC(한국 Tier-1)과 자율주행 트랙터를 공동개발하고, 2026년부터 양산 판매를 시작하며, 2030년 $35M 매출을 전망하는 Agricultural Robotics 회사입니다.

SS500에서 경험한 것은: - 모터 제어 → 물리적 액추에이터를 정밀하게 움직이는 기술 - CAN 분산 제어 → 다수의 노드가 실시간으로 협조하는 시스템 - 상태머신 + 기능안전 → 사람 근처에서 동작하는 기계의 안전 설계 - 배터리 + 전력 관리 → 전동화 제품의 에너지 시스템 - 자율주행 통합 → AI가 물리 세계를 인지하고 행동하는 루프

이 모든 것은 로보틱스, 자율주행 차량, 스마트 디바이스 — Physical AI가 확장되는 모든 영역의 기반 기술입니다.

엔지니어링 철학

일론 머스크의 개발 5단계 프로세스를 실무에 적용합니다:

단계 원칙 SS500 적용 사례
1. 요구사항을 의심하라 주어진 스펙을 그대로 받지 않는다 아세아텍 100A 벤치마킹 기준이 우리 타겟 고객에 맞는지 재검토
2. 삭제하라 불필요한 것을 제거한다 제어권 NONE 상태를 삭제하여 상태머신 복잡도 감소 + 안전 버그 해결
3. 단순화/최적화하라 남은 것을 더 단순하게 CAN 메시지 표준화, 제어권 전환 시 Init 루틴 통일
4. 속도를 높여라 최적화된 것을 빠르게 배터리 정합성 검증 자동화 체크리스트 도입
5. 자동화하라 반복 작업을 자동화 Claude Code로 이슈 관리, 주간보고, 기술 분석 자동화

지향점

Physical AI 제품의 하드웨어 개발 — 로봇, 자율주행, 스마트 디바이스

하드웨어를 이해하면서 소프트웨어도 쓸 수 있는 브릿지 엔지니어로서, AI가 물리 세계와 만나는 접점에서 "실제로 동작하는 제품"을 만들고 싶습니다.

정량적 근거: HIH_2 프로젝트 원본 + GINT 공개 자료 기반

Case Study: 배터리 BMS 정합성 검증

프로젝트: PLUVA FRIEND SS500 (GT-SS500) 역할: 전력제어개발팀 선임연구원 (이슈 발행 및 검증 주도) 기간: 2025.12 ~ 2026.02 (3개월, 진행 중) 이슈 번호: ISSUE #72 (최우선), #78, #000

Situation — 상황

SS500은 48V 460Ah LiFePO4 배터리(22.08kWh, 15S2P 구성)를 탑재한 전동 농기계입니다. 배터리는 외부 업체가 공급하며, VCU(Vehicle Control Unit)와 CAN 버스로 통신하여 SOC(State of Charge), 셀 전압, 온도, 보호 상태를 실시간 모니터링합니다.

Alpha 3호기 + Pilot 2호기가 제작된 시점에서, 배터리 BMS의 CAN 정합성과 보호 로직 검증이 양산 전 필수 관문이었습니다.

왜 이게 중요한가: - 48V 460Ah는 상당한 에너지(22kWh) — 보호 로직 미작동 시 화재/폭발 위험 - 농기계는 비전문가(농민)가 운용 — 경고 표시가 정확해야 함 - 양산 16대 + Spare 4대가 3월 출하 예정 — 배터리 품질이 양산 일정을 좌우

Task — 과제

핵심 질문: "배터리 업체가 납품한 BMS의 보호 로직이 실제로 동작하는가?"

구체적 검증 항목: 1. CAN DB 정합성: 배터리 업체가 정의한 CAN 신호가 우리 VCU와 맞는가? 2. SOC 정확도: 충전/방전 시 SOC가 정확하게 표시되는가? 3. 보호 로직: 과충전/과방전/셀 불균형 시 Warning → Fault → Protection이 순차 발동하는가? 4. Kill Switch: 비상정지 시 배터리가 안전하게 Shutdown되는가? 5. Power Hold: VCU가 Shutdown 요청 시 릴레이가 정상 차단되는가?

제약 조건: - 배터리 업체와 물리적 거리 (합동 시험이 쉽지 않음) - ROM 업데이트가 무선이 아닌 현장 방문 필요 - Alpha 1~3호기 각각 다른 ROM 버전이 적용된 상태

Action — 행동

1차 검증 (2025.12.10) — CAN 정합성 기본 확인

방법: Alpha 1호기 납품 배터리의 CAN 로그를 Vector 장비로 수집하여 분석

발견한 문제: - CAN DB가 신규 버전으로 업데이트되었으나, Alpha 1호기에만 적용됨 - 나머지 호기(Alpha 2,3 / Pilot 2~4)는 구형 ROM(Ver 10) 유지 - 배터리 업체 제공 CAN DB와 실측 신호 불일치 항목 존재

산출물: 20251210_배터리_및_충전기_정합성_검토_결과.xlsx

2차 검증 (2026.01.14) — 보호 로직 심층 검증

방법: Alpha 1호기 ROM 업데이트 후, 완충→방전 전 구간 CAN 로그 수집 + 셀별 전압 분석

면접 예상 Q&A

Q: 배터리 업체와 갈등은 없었나요?

ROM 업데이트 3차까지 반복하면 업체 관계가 불편해질 수 있습니다. 저는 "왜 안 되냐"가 아니라 "어떤 신호가 기대값과 다르고, 어떤 조건에서 재현된다"는 데이터를 매번 정리해서 전달했습니다. 업체 입장에서도 모호한 클레임보다 명확한 버그 리포트가 더 도움이 됩니다. 결과적으로 WBS를 공동으로 작성하는 수준까지 협력 관계를 발전시켰습니다.

Q: LiFePO4인데 열폭주 걱정이 과한 거 아닌가요?

LiFePO4는 NMC 대비 열적 안정성이 높은 건 맞습니다. 하지만 460Ah 대용량 팩에서 셀 불균형(260mV)이 누적되면, 약한 셀에 과부하가 집중되어 수명 급감 → 내부 단락 위험이 있습니다. 화학적으로 안전한 셀이라도, 시스템 레벨에서 보호 로직은 반드시 검증해야 합니다. 이건 "혹시 몰라서"가 아니라 "기본 안전 설계 원칙"입니다.

Q: 이 경험을 다른 분야에 어떻게 적용할 수 있나요?

배터리 BMS 정합성 검증의 핵심은 "외부 모듈과 CAN으로 통신하는 시스템에서, 기대 동작과 실제 동작의 차이를 체계적으로 검증하는 것"입니다. 이건 배터리뿐 아니라, 모터 컨트롤러, 센서 모듈, 자율주행 PC — CAN 기반 분산 시스템이라면 어디든 적용됩니다. 특히 로보틱스나 자율주행 차량에서 다수의 ECU가 통신하는 구조에서 이 검증 역량은 직접적으로 가치가 있습니다.

Case Study: 제어권 상태머신 — 안전한 모드 전환의 설계

프로젝트: PLUVA FRIEND SS500 (GT-SS500) 역할: 전력제어개발팀 선임연구원 (상태머신 로직 검토/설계/검증) 기간: 2026.01 ~ 2026.02 이슈 번호: ISSUE #79

Situation

SS500은 3가지 제어 모드를 지원합니다:

모드 입력 장치 사용 상황
RC RF 리모콘 근거리 수동 조종 (주 사용 모드)
LCD 본체 디스플레이 패널 현장 세팅, 파라미터 조정
ADT 자율주행 PC (GNSS+카메라+IMU) 자동 경로 추종 방제

문제는 이 모드들 사이를 전환할 때 안전을 어떻게 보장하는가입니다.

농기계는 비전문가(농민)가 운용합니다. RC로 조종하다가 LCD로 바꾸거나, 자율주행 중에 리모콘 페어링이 되거나, 비상정지 후 복귀할 때 — 모든 전환에서 모터가 예상대로 동작해야 합니다. 예상과 다르게 동작하면 사고입니다.

Task

핵심 질문: "어떤 모드에서 어떤 모드로 전환하든, 차량이 예측 가능하게 동작하는가?"

구체적 검증 항목: 1. RC → LCD 전환 시 구동모터 속도가 초기화되는가? 2. LCD → ADT 전환 시 자율주행이 정상 시작되는가? 3. ADT → RC 강제 전환 시 기존 자율주행 속도가 잔존하지 않는가? 4. 비상정지 후 모든 모터 상태가 안전하게 초기화되는가?

Action

1단계: DM 로직 및 하위 제어기 상태머신 알고리즘 검토 (~01/15)

VCU의 DM(Decision Maker) 로직과 각 모터 컨트롤러의 상태머신을 코드 레벨에서 리뷰했습니다.

발견한 치명적 버그:

RC 모드에서 속도 3km/h로 주행 중, 리모콘 페어링이 발생하면 제어권이 RC→LCD로 전환됩니다. 이때 LCD 모드에서는 속도 지령을 보내지 않기 때문에, 모터 컨트롤러의 상태머신에 이전 속도(3km/h)가 그대로 남아 있었습니다.

차량이 LCD 모드로 전환되었는데 아무 조작 없이 계속 전진하는 상황.

근본 원인 분석:

RC 모드 (속도 3km/h 주행 중)
    ↓ 리모콘 페어링 발생
LCD 모드 진입
    ↓ LCD에서 속도 지령을 보내지 않음
모터 컨트롤러: "새 지령이 없으니 마지막 지령(3km/h) 유지"
    ↓
차량이 무인 상태로 계속 전진 ← 위험!

더 깊은 문제: 기존 설계에 CTRL = NONE 상태가 존재했습니다. "제어권 없음" 상태인데, 이 상태에서는 어떤 모듈도 모터를 제어하지 않아 마지막 속도가 유지됩니다.

2단계: 아키텍처 변경 설계 (~01/23)

두 가지 핵심 변경을 제안했습니다:

변경 1 — NONE 상태 삭제

Before: RC ↔ NONE ↔ LCD ↔ NONE ↔ ADT
After:  RC ↔ LCD ↔ ADT (항상 명시적 제어 모드)

NONE 상태를 삭제하면 "아무도 제어하지 않는" 공백이 사라집니다.

변경 2 — 모든 Transition에서 Init 루틴 적용

제어권 전환 감지
    ↓
모든 모터 컨트롤러에 속도 0(Init) 지령 인가
    ↓
새 제어 모드 활성화
    ↓
새 모드에서 속도 지령 시작

LCD 제어권일 때는 속도 0을 지속적으로 인가하여, 과거 속도가 잔존하지 못하도록 했습니다.

Situation

압력은 우위지만, 유량이 4배 이상 부족해서 넓은 과수원에서의 방제 효율이 떨어진다는 현장 피드백이 있었습니다.

Task

핵심 질문: "펌프를 업그레이드해야 하는가? 한다면 어디까지?"

단순히 "더 좋은 펌프를 달면 된다"가 아닙니다. 전동 시스템에서 펌프를 키우면: - 전류가 늘어나고 - 배터리 소모가 빨라지고 - 운전시간이 줄어들고 - 비용이 올라갑니다

이 트레이드오프를 정량적으로 분석해서 경영진이 의사결정할 수 있는 데이터를 제공하는 것이 과제였습니다.

Action

1단계: 전력 예산 분석

노즐 1.5mm × 19개 (아세아텍 100A 수준):

펌프 수동력 = P × Q = 1.5MPa × 66LPM = 1.65 kW
모터 입력 = 1.65 / 0.51(효율) = 3.23 kW (4.3마력)
필요 전류 = 3.23kW / 50V = 64.0 A

현재 SS500 (노즐 1.0mm × 12개):

펌프 수동력 = 2.0MPa × 22.58LPM = 0.75 kW
모터 입력 = 0.75 / 0.55 = 1.36 kW (1.8마력)
필요 전류 = 1.36kW / 50V = 27.0 A

Case Study: 확인하고 싶으면 장비를 만들어서라도 본다

"측정할 수 없으면 개선할 수 없다" — 피터 드러커 "시험 방법이 없으면 만든다. 데이터가 없으면 만든다. 감으로 하지 않는다."

마인드셋

엔지니어링에서 가장 위험한 말은 "대충 될 것 같다"입니다.

저는 확인하고 싶은 것이 있으면, 시험 방법이 없어도 장비를 만들어서라도 봅니다. 이것은 취미가 아니라 원칙입니다. 3년간 9개의 시험 체계를 직접 설계/구축/운영하면서, 매번 같은 루프를 반복했습니다:

가설 수립 → 측정 방법 설계 → 장비/벤치 구축 → 데이터 수집 → 정량적 결론 → 의사결정

아래는 시간순으로 엮은 그 여정입니다.

Phase 1. 기초를 쌓다 (2023)

1. 다이나모미터 토크제어 시스템 — "토크를 전압으로 변환하다"

가설: 모터 성능 평가를 위해 정밀한 토크 측정이 필요한데, 기존 다이나모미터의 토크 제어가 불안정하다.

만든 것: - Strain Gauge 기반 Full Bridge Circuit 설계 - Wheatstone Bridge 아날로그 신호 처리 회로 - 토크 컨트롤러 펌웨어 개발 - 10포인트 교정 체계 (0~9.811 dN·m)

정량적 결과: | 항목 | 수치 | |------|------| | 토크 감도 | 0.996 N·m | | 비선형성 | 0.008% | | 교정 포인트 | 10개 (완벽 교정) | | 추가 성과 | 사내 기술 세미나 개최 (질의응답 포함) |

임팩트: 이후 EOP 400W, ADT 모터의 모든 성능 시험에서 이 다이나모미터가 표준 시험 장비로 사용됨. "만든 장비가 회사의 인프라가 된" 첫 사례.

Phase 2. 방법론을 확장하다 (2024)

2. PWM 절환 온도 비교 시험 — "16°C의 차이를 숫자로 증명하다"

가설: SVPWM이 기존 DPWM 대비 FET 온도가 낮을 것이다. 하지만 "낮다"만으로는 부족하다. 얼마나 낮은지 정량화해야 한다.

만든 것: - SVPWM vs DPWM 비교 시험 프로토콜 설계 - 온도 조건별(55°/85°/120°) 시험 매트릭스 - 실시간 데이터 로깅 체계 (CSV 출력)

정량적 결과: | 항목 | 수치 | |------|------| | 데이터 포인트 | 2,932개 (48분 52초 연속 수집) | | FET 온도 저감 | 16~17°C (SVPWM 적용 시) | | 시험 조건 | 55°/85°/120° × 2개 Assy × 2개 PWM 방식 = 12세트 |

임팩트: 국책과제(EOP 400W) 완료보고서의 핵심 근거 데이터. "SVPWM 적용이 열 성능을 개선한다"는 주장을 12세트 비교 시험으로 입증.

3. ADT 모터 실주행 비교 — "벤치가 아닌 실제 도로에서 증명하다"

가설: 3세대 모터 개선(TVS Diode 적용, PI 파라미터 최적화)이 실제 주행에서도 효과가 있을 것이다.

만든 것: - 실주행 데이터 수집 → 개선 전/후 비교 분석 프레임워크 - Zbench 지표 기반 정량 평가 체계

정량적 결과: | 항목 | Before | After | 개선 | |------|--------|-------|------| | Zbench 지표 | 2.714 | 4.346 | +60% | | 오버슈팅(OVT) | 발생 | 제거 (TVS Diode) | 100% | | 리플 안정성 | 39~147 ppm (변동 큼) | 139~142 ppm | 안정화 |

Phase 3. 시스템으로 진화하다 (2025~2026, SS500)

SS500 프로젝트에서는 단일 시험이 아니라, 제품 전체를 커버하는 시험 체계를 구축했습니다.

4. 펌프 성능평가 시스템 — "상용 시험기가 없으면 직접 만든다"

가설: 펌프 3개 시료 중 어떤 것이 최적인지, 그리고 압력-유량 관계가 선형인지 확인해야 한다. 하지만 48V 전동 펌프용 성능평가 시스템은 시장에 없다.

만든 것: - Blancett B2800 유량계 + SENSYS PSH 압력계 조합 - 배관 + 밸브 + 탱크 + 계측 구성 - 테스트베드 단독 구축 (ISSUE #126) - 시험 일정: 구축(~02/14) → 시운전(~02/21)

정량적 결과: | 시료 | 유량 (L/min) | 비고 | |------|-------------|------| | 시료 A | 10.4 | 최저 | | 시료 B | 13.0 | 중간 | | 시료 C | 15.6 | 최고 | | 발견 | 압력-유량 선형관계 규명 | 펌프 선정 기준 수립 |

임팩트: 3개 시료의 성능 차이를 정량적으로 비교하여 펌프 선정 의사결정의 근거를 마련. "감성 평가"가 아닌 "데이터 기반 선정".

5. 팬 시스템 벤치 — "80% 갭 축소의 효과를 숫자로 보여주다"

가설: 프로펠러 크기를 키우고(18→20파이) 슈라우드 갭을 줄이면(45→9mm) 풍량이 증가할 것이다. 하지만 "체감상 세다"로는 부족하다.

만든 것: - 카메라 스탠드 + 클램프 + 프로펠러형 풍속계(~50m/s) 조합 - 거리별(1, 2, 3, 4, 5m) 풍속 측정 체계 - RS-500X(기계식 기성 모델) 대조군 포함 - 감성평가 + 정량평가 동시 수행 프로토콜

정량적 결과: | 항목 | Before | After | 변화 | |------|--------|-------|------| | 프로펠러 직경 | 18파이 | 20파이 | +11% | | 슈라우드 갭 | 45 mm | 9 mm | -80% | | 모터 출력 | 19.7 W | 42.2 W | +114% | | 모터 전류 | 16.74 A | 38.7 A | +131% | | 감성평가 | - | "풍압 증가 감지됨" | 정성 확인 |

임팩트: 갭 축소 설계 변경의 효과를 Before/After로 입증. 기구팀에 모터 플레이트 변경(4t→12t) + 어댑터(5t) 추가 설계를 정량적 근거와 함께 요청.

6. 범퍼 안전 시험 — "0.082m를 숫자로 증명하다"

가설: 범퍼 감지 시 차량이 충분히 빠르게 정지하는가? 자율주행 안전의 핵심.

만든 것: - 속도별 정지거리 6단계 이론 모델 수립 (0.5~3.0 km/h) - 물리 공식: 정지거리 = v² / (2a), a = 0.5167 m/s² - 범퍼 스트로크(123.5mm), 센서 인식거리(17.5mm) 기반 설계 사상 정의 - 실차 시험 → 이론값 vs 실측값 비교

정량적 결과: | 속도 (km/h) | 이론 정지거리 (m) | 실측 정지거리 (m) | 정지시간 | 피크 전류 | |------------|-----------------|-----------------|---------|----------| | 3.0 | 0.672 | 0.082 | ≤0.05초 | 308 A | | 2.0 | 0.299 | 0.055 | - | 158.8 A | | 1.0 | 0.075 | 0.028 | - | -156.5 A |

핵심 발견: 이론 모델(등가속 가정)은 실측 대비 8배 이상 보수적. 실제로는 전기 제동(308A 피크)이 기계적 관성보다 훨씬 빠르게 작용. 브레이크 전류 정착시간 467ms도 정량화.

임팩트: 자율주행 안전 인증의 핵심 데이터. "범퍼가 있다"가 아니라 "3km/h에서 0.082m 내에 정지한다"고 말할 수 있게 됨.

7. SS500 vs 기성모델 성능 비교 — "경쟁 제품을 빌려서라도 비교한다"

가설: SS500이 기계식 기성모델(한아 SS기) 대비 어떤 점에서 우위/열위인지 정량적으로 파악해야 한다.

만든 것: - 기성모델 실물 대여 (대여기간 ~01/08) - 노즐 시스템, 펌프 시스템, 팬 시스템 3대 축 비교 프레임워크 - 분사 성능 정성적 비교: 노즐 18ea vs 36ea 실제 분사 영상 촬영

정량적 결과: | 항목 | SS500 | 기성모델(한아) | 비교 | |------|-------|-------------|------| | 펌프 압력 | 30 bar | 살포 15 / 급수 25 bar | SS500 2배 | | 펌프 유량 | 24 L/min | 99 L/min | 기성모델 4배 | | 노즐 수 | 12 | 36 | 기성모델 3배 | | 송풍 회전수 | ~8000 rpm | ~3000 rpm | SS500 2.7배 | | 사용자 제어성 | 3단계 + 개별 제어 | 로터리 + 연동 | SS500 우위 |

8. 배터리 CAN 정합성 검증 체크리스트 — "신뢰하되, 검증하라"

가설: 배터리 업체가 "수정 완료"라고 한 ROM이 실제로 정합성을 만족하는가?

만든 것: - 신호별 기대값 vs 실측값 자동 비교 시트 - 검증 체크리스트 (CAN DB / Kill Switch / Power Hold / SOC / Warning-Fault-Protection) - 3차 반복 검증 프로토콜 (25.12 → 26.01 → 26.02)

9. 내구성 시험 설계 — "128,000 사이클을 견딜 수 있는가"

가설: 양산 전에 SS500의 핵심 부품(펌프, 솔레노이드, 교반 시스템)이 실사용 조건에서 충분한 내구성을 확보해야 한다.

만든 것: - 128,000 Cycle Kill Test 시험 계획 수립 - Worst Case 교반 Profile 정의 - 극한 조건 매트릭스: 온도(고온/저온) × 습도(고습/건조) × 지형(평지/경사) - 내구 시험 검토서 작성 (20251209)

임팩트: 양산 품질 확보의 기초. "프로토타입이 동작한다"와 "양산품이 3년 쓴다"는 전혀 다른 문제이며, 그 차이를 메우는 것이 내구성 시험.

면접 예상 Q&A

Q1. 시험장비를 직접 만드는 건 비효율적이지 않나요? 외주를 주면 되잖아요.

48V 전동 농기계용 펌프 성능평가 시스템은 시장에 존재하지 않습니다. 외주를 줘도 우리 제품의 운전 조건(48V, 10~40LPM, CAN 연동)을 이해하는 업체를 찾기 어렵고, 찾더라도 리드타임이 수개월입니다. 제가 2주 만에 구축한 펌프 벤치가 외주 대비 빠르고, 정확하고, 반복 사용 가능했습니다.

더 중요한 건, 시험 장비를 직접 만들면 "무엇을 측정해야 하는지"에 대한 이해가 깊어진다는 것입니다. 다이나모미터를 만들면서 토크 측정의 원리를 체득했고, 그 이해가 이후 모든 모터 평가에 적용됐습니다.

Q2. 가장 만족스러운 시험 결과는?

범퍼 안전 시험입니다. 이론 모델에서는 3km/h 정지거리가 0.672m로 계산됐는데, 실측은 0.082m — 8배 이상 좋았습니다. 이론(등가속 가정)과 실제(전기 제동)의 차이를 정량적으로 확인한 순간이었고, 동시에 "전동 구동의 강점이 여기에 있다"는 확신을 얻었습니다. 308A 피크 전류로 0.05초 만에 정지하는 전기 제동은 기계식 브레이크와 차원이 다릅니다.

Q3. 데이터가 기대와 다를 때 어떻게 하나요?

SOC 162% 사례가 정확히 그런 경우입니다. "보호 로직이 동작해야 한다"는 기대와 "아무런 Warning도 발생하지 않는다"는 실측이 완전히 반대였습니다. 이때 중요한 건 데이터를 의심하지 않는 것입니다. "장비가 잘못된 건 아닌가?" 대신 "왜 이 값이 나왔는가?"를 먼저 물어봅니다.

SOC 162%의 경우, 셀별 전압(7번 셀 3.34V vs 평균 3.46V)까지 파고 들어가서 BMS의 SOC 계산 로직 자체에 문제가 있음을 확인했습니다. 데이터를 믿고, 데이터의 원인을 추적하는 것이 원칙입니다.
